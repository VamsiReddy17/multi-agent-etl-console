# Kafka Messaging Guide

Apache Kafka serves as the central high-throughput streaming message broker, decoupling event generation from our downstream processing stages.

---

## 🏗️ High-Level Kafka Event Architecture

```
                                 ┌────────────────────────────────────────────────────────┐
                                 │                Local PC (Docker Host)                  │
                                 │                                                        │
┌────────────────┐               │  ┌──────────────────────┐      ┌────────────────────┐  │
│ Order Generator│ ──(Publish)──►│  │ Topic: orders        │ ───► │ Ingestion Agent    │  │
│ (Mock Engine)  │               │  └──────────────────────┘      │ (Polls batched docs│  │
└────────────────┘               │                                │  up to 2000 items) │  │
                                 │  ┌──────────────────────┐      └────────────────────┘  │
                                 │  │ Topic: dead_letter   │ ◄─── ┌────────────────────┐  │
                                 │  │ (Quarantined events) │      │ DeadLetterAgent    │  │
                                 │  └──────────────────────┘      │ (DLQ Router)       │  │
                                 │                                └────────────────────┘  │
                                 └────────────────────────────────────────────────────────┘
```

## ⚙️ Active Topics & Message Flows

### 1. Topic: `orders`
* **Role**: Receives real-time JSON transactional event payloads generated by mock databases or CRM APIs.
* **Schema Contract**:
  ```json
  {
    "order_id": 1,
    "customer_id": 101,
    "product_id": 5,
    "quantity": 2,
    "amount": 49.99,
    "event_type": "order_placed"
  }
  ```

### 2. Topic: `dead_letter`
* **Role**: Acts as our Dead Letter Queue (DLQ). If a record is rejected by the `QualityAgent` (e.g. negative quantity, missing fields, or duplicate transaction IDs), the `DeadLetterAgent` captures the bad payload, stamps an `error_message`, and writes it here for safe isolation.
* **Benefit**: Downstream loader tasks can continue processing clean batches without getting blocked by structural schema violations or corrupted payloads.

## 🛠️ CLI Operations

### Listing active topics
```bash
docker exec prod_kafka kafka-topics --list --bootstrap-server localhost:9092
```

### Inspecting Consumer Group Offset Lag
To check the processing status and offset positions of your active consumers:
```bash
docker exec prod_kafka kafka-consumer-groups \
  --bootstrap-server localhost:9092 \
  --describe \
  --group prod-pipeline-group
```
This prints the `CURRENT-OFFSET`, `LOG-END-OFFSET`, and `LAG` for all partitions, helping you monitor real-time ingestion performance.

### Producing Test Payloads manually
```bash
docker exec -it prod_kafka kafka-console-producer --broker-list localhost:9092 --topic orders
```
Paste a JSON record and press **Enter**:
```json
{"order_id": 9999, "customer_id": 3, "product_id": 1, "quantity": 1, "amount": 100.0, "event_type": "order_placed"}
```

### Resetting Consumer Offsets
To force-reprocess all messages in a topic (replaying history):
```bash
docker exec prod_kafka kafka-consumer-groups \
  --bootstrap-server localhost:9092 \
  --group prod-pipeline-group \
  --topic orders \
  --reset-offsets \
  --to-earliest \
  --execute
```
